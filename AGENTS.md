# AGENTS

Инструкции для агентных ассистентов в этом репозитории.

## Контекст проекта

* Единый Python-проект `codelab/`, объединяющий сервер и клиент ACP.
* Менеджер окружения и запуск команд: `uv`.
* Базовые проверки запускаются через `Makefile` из корня репозитория.
* Основной язык документации: русский.
* Минимальная поддерживаемая версия Python: 3.12.

---

## Приоритет правил

При конфликте требований использовать следующий порядок:

1. Корректность работы системы.
2. Соответствие ACP, A2A и MCP спецификациям.
3. Сохранение обратной совместимости.
4. Минимальность изменений.
5. Следование существующему стилю проекта.

Если для исправления ошибки необходимо нарушить обратную совместимость, изменение должно быть явно отмечено как изменение контракта и сопровождаться проверкой совместимости.

## Правила работы с кодом

### Создание новых абстракций

Перед созданием нового файла, класса, сервиса, менеджера, фабрики, реестра или обработчика необходимо:

1. Проверить существующие реализации.
2. Проверить возможность расширения существующих компонентов.
3. Проверить отсутствие аналогичной абстракции.
4. Создавать новую абстракцию только при наличии объективной необходимости.

Новая абстракция должна быть исключением, а не правилом.

### Повторное использование кода

Перед созданием новой логики:

1. Проверить существующие реализации.
2. Переиспользовать существующие компоненты при возможности.
3. Избегать дублирования бизнес-логики.
4. Предпочитать композицию копированию кода.

### Типизация

* Использовать современную типизацию Python 3.12+.
* Не ослаблять существующие типы без необходимости.
* Не использовать `Any`, если можно указать конкретный тип.
* Не использовать `# type: ignore`, кроме случаев, когда это документировано и обосновано.

### Ошибки и логирование

* Не скрывать ошибки.
* Не использовать `except Exception: pass`.
* Сохранять исходную причину ошибки через `raise ... from ...`, если исключение преобразуется.
* Использовать существующую инфраструктуру логирования проекта.
* Новые точки отказа должны сопровождаться понятными сообщениями об ошибках.

### Асинхронный код

* Не блокировать event loop.
* Не использовать синхронный ввод-вывод внутри async-кода без необходимости.
* Предпочитать существующие async-абстракции проекта.
* Корректно обрабатывать отмену задач (`asyncio.CancelledError`).
* Не создавать фоновые задачи без контроля их жизненного цикла.

### Комментарии и документация к коду

* Предпочитать понятные имена функций, методов и переменных комментариям.
* Комментарии добавлять только там, где логика неочевидна.
* Запрещены комментарии, пересказывающие код.
* Публичные API документировать через docstring.
* Не оставлять закомментированный код в репозитории.

Пример плохого комментария:

```python
# Добавляем сообщение в список
messages.append(message)
```

Пример хорошего комментария:

```python
# ACP требует сохранять исходный порядок сообщений для корректного воспроизведения истории.
messages.append(message)
```

## Тестирование

### Обязательные требования

Каждое изменение поведения должно сопровождаться тестами.

Необходимо:

* добавлять новые тесты для новой функциональности;
* обновлять существующие тесты при изменении поведения;
* исправлять сломанные тесты только если изменение поведения является ожидаемым.

### Источник истины

Существующие тесты считаются источником истины о текущем поведении системы.

Перед изменением реализации необходимо изучить тесты соответствующего модуля.

### Виды тестов

Использовать подходящий уровень тестирования:

* unit-тесты;
* интеграционные тесты;
* сквозные тесты при необходимости.

## Протоколы и совместимость

### Запрещено изменять

Никогда не изменять документацию:

* `doc/Agent Client Protocol/`
* `doc/Agent To Agent Protocol/`
* `doc/Model Context Protocol/`

Это официальные внешние спецификации.

### Запрещено нарушать

Все изменения в коде обязаны соответствовать спецификациям:

* Agent Client Protocol (ACP)
* Agent To Agent Protocol (A2A)
* Model Context Protocol (MCP)

### Изменения, считающиеся изменением протокола

Следующие изменения требуют отдельной проверки совместимости:

* JSON-RPC сообщения;
* ACP Content Types;
* Session State;
* Tool Call State;
* публичные ACP API;
* MCP API;
* форматы сериализации;
* transport layer контракты.

## Архитектурные ограничения

### Клиентская архитектура

Клиент реализован по принципам Clean Architecture.

Допустимые зависимости:

```text
domain
↑
application
↑
infrastructure
↑
presentation
↑
tui
```

Внутренние слои не должны зависеть от внешних.

Запрещены зависимости:

* `domain -> application`
* `domain -> infrastructure`
* `domain -> presentation`
* `domain -> tui`
* `application -> infrastructure`
* `application -> presentation`
* `application -> tui`

### Серверная архитектура

* Не обходить ACP Protocol Layer.
* Не обращаться напрямую к внутреннему состоянию сессий, если существует специализированный менеджер.
* Использовать существующие Registry, Factory и Manager компоненты.
* Соблюдать существующие границы между protocol, storage, agent, tools и llm.

## Технические решения

### MCP SSE Transport

`SseTransport` помечен как deprecated в MCP specification.

Удалять его запрещено.

Причина:

* используется существующими клиентами;
* необходим для обратной совместимости.

Любые изменения должны сохранять поддержку текущего поведения.
