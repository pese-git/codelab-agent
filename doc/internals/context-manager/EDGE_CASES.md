## Кейс 1 — Пустой файл / файл из пробелов

**Ситуация.** `ContextGatherer.gather()` через ACP `read_file()` получает файл с пустым
содержимым (`""`) или из одних пробелов/переводов строк. Далее путь идёт в
`CodeSkeletonizer.skeletonize()` и в подсчёт `TokenCounter.count()`.

**Ожидаемое поведение.**
- Файл **не отбрасывается молча**, но и не создаёт «пустой» `ContextItem` с нулевой пользой:
  пустое / whitespace-only содержимое после `str.strip()` → элемент **не добавляется** в
  результат `gather()` (нет смысла занимать слот в `baseline`).
- Если такой файл всё же попадает в компактор (`ContextCompactor`), `CodeSkeletonizer.skeletonize("")`
  возвращает вход без изменений (детерминированно), а `TokenCounter.count("")` → `0`.
- `FileContentCache.set(path, "")` допустим: пустота — валидное содержимое, кэш-хит на него
  легитимен (повторный `read_file` не нужен).

**Какой слой/компонент отвечает.** Слой A — `ContextGatherer` (фильтрация пустых при отборе);
слой C — `CodeSkeletonizer`, `TokenCounter`, `FileContentCache` (корректная обработка пустой строки).

**Acceptance.**
- `gather()` для каталога, где целевой файл пуст / только пробелы, не возвращает для него `ContextItem`.
- `TokenCounter.count("")` == `0`; `TokenCounter.count("   \n\t")` >= `0` без исключений.
- `CodeSkeletonizer.skeletonize("")` == `""`; повторный вызов даёт тот же результат (детерминизм).
- `FileContentCache.get(path)` после `set(path, "")` возвращает `""`, **не** `None` (различение «нет в кэше» и «пусто»).

## Кейс 2 — Бинарный / нетекстовый файл при read

**Ситуация.** Под отбор `ContextGatherer` попадает `.png`, `.zip`, `.pdf` или текстовый файл
с битой UTF-8-последовательностью. `read_file()` возвращает байты, которые не декодируются как текст.

**Ожидаемое поведение.**
- Декодирование выполняется на границе ACP `ToolRegistry`; при `UnicodeDecodeError` `ContextGatherer`
  ловит ошибку и **исключает** файл из отбора (он не пригоден как `ContextType.FILE_CONTENT`).
- Бинарь **не** уходит в `FileContentCache.set()` и **не** в `CodeSkeletonizer` (AST-парсинг сорвётся).
- Решение фиксируется в логе (`binary_file_skipped` с путём), без падения хода.
- Эвристика отсечения до чтения: расширение из bin-чёрного списка → файл не читается вовсе
  (экономия ACP RPC), и `DependencyGraph` его не учитывает как узел исходника.

**Какой слой/компонент отвечает.** Слой A — `ContextGatherer` (отлов ошибки декодирования, фильтр
по расширению); слой C — `FileContentCache` / `CodeSkeletonizer` не вызываются для бинаря.

**Acceptance.**
- `gather()` по каталогу с бинарным файлом завершается без исключений и не содержит его в результате.
- `FileContentCache.get(<bin_path>)` == `None` (не закэширован).
- Лог содержит запись о пропуске бинаря.
- `CodeSkeletonizer.skeletonize` не вызывается для бинарного содержимого (проверяется моком/счётчиком вызовов).

## Кейс 3 — Файл больше всего бюджета окна

**Ситуация.** Один файл оценивается, например, в 200 000 токенов при
`ContextConfig.max_context_tokens = 128000`. То есть он не помещается даже целиком в окно,
не говоря о доле истории (`history_share`).

**Ожидаемое поведение.** Двухступенчатая деградация, **система не падает**:
1. `TokenBudgetManager.allocate()` даёт долю под файл; `TokenBudgetManager.bound_content(content, max_tokens)`
   усекает содержимое, **сохраняя начало и конец** (сигнатуры/импорты сверху, обычно важное снизу).
2. Если файл — код, перед усечением применяется `CodeSkeletonizer.skeletonize()` (фаза Skeletonize):
   тела функций → `...`, остаются сигнатуры. Скелет + `bound_content` радикально снижают объём.
- Результирующий `ContextItem.content` помечается как усечённый (тип может стать `FILE_SKELETON`),
  чтобы LLM понимала, что видит не весь файл.
- Это graceful degradation (MVP-выбор аналогичен FCM Option A: truncate, не reject).

**Какой слой/компонент отвечает.** Слой A — `TokenBudgetManager.bound_content()`;
слой C — `CodeSkeletonizer` (предварительное скелетирование), `TokenCounter` (оценка до/после).

**Acceptance.**
- Для файла > `max_context_tokens` итоговый `ContextItem.token_count <= allocate().history_tokens` (или иной выделенной доли).
- `bound_content(huge, N)` возвращает строку, в которой присутствуют и первые, и последние строки исходника.
- Для огромного `.py`: сначала пробуется скелет; если скелет всё ещё велик — добивается `bound_content`.
- Ход завершается без исключения переполнения.

## Кейс 4 — Файл не на Python (can_handle=False)

**Ситуация.** В контекст попадает `config.json`, `Dockerfile`, `README.md`, `widget.dart`.
`PythonASTSkeletonizer` не умеет их парсить.

**Ожидаемое поведение.**
- `CodeSkeletonizer.can_handle(path)` для неподдерживаемого расширения возвращает `False`.
- Компактор (фаза Skeletonize) **пропускает** скелетирование и кладёт содержимое **как есть**
  (с поправкой на `bound_content`, если велико — см. Кейс 3).
- Никаких «частичных» AST-результатов: либо полноценный детерминированный скелет, либо оригинал.
- Расширение через новые реализации `CodeSkeletonizer` (JS/TS/Dart) — post-MVP, по Open/Closed,
  без изменения сигнатур ABC.

**Какой слой/компонент отвечает.** Слой C — `CodeSkeletonizer.can_handle()` (gate);
`ContextCompactor` (ветвление: скелет vs оригинал).

**Acceptance.**
- `PythonASTSkeletonizer.can_handle("config.json")` == `False`; `can_handle("foo.py")` == `True`.
- Компактор для `.json` оставляет содержимое идентичным входу (байт-в-байт, важно для baseline-стабильности).
- `skeletonize()` не вызывается, когда `can_handle()` == `False` (проверяется моком).

## Кейс 5 — Циклические импорты в DependencyGraph

**Ситуация.** `a.py` импортирует `b.py`, `b.py` импортирует `a.py` (или более длинный цикл).
`ContextGatherer` строит граф и вызывает `get_dependencies(path, recursive=True)`.

**Ожидаемое поведение.**
- `DependencyGraph.get_dependencies(recursive=True)` обходит граф с **множеством посещённых узлов**;
  встретив уже посещённый узел — не входит в него повторно. **Нет бесконечной рекурсии / переполнения стека.**
- Каждый файл цикла попадает в результат **ровно один раз** (дедуп на уровне обхода).
- Порядок результата детерминирован (например, по порядку первого посещения) — важно для стабильности baseline.

**Какой слой/компонент отвечает.** Слой A — `DependencyGraph.get_dependencies()` (защита от циклов);
`ContextGatherer` (использует результат для отбора без дублей).

**Acceptance.**
- Граф с циклом `a→b→a`: `get_dependencies("a", recursive=True)` завершается, возвращает `["b"]` (или `["b","a"]` по правилу), без повторов.
- Тест на цикл длины 3 (`a→b→c→a`) завершается за конечное время.
- Повторный вызов даёт идентичный по порядку результат (детерминизм).

## Кейс 6 — Файл изменён во время хода (гонка fs/write ↔ reconcile)

**Ситуация.** Между `build_context()` (baseline уже собран по версии файла V1) и моментом, когда
`ContextReconciler.reconcile()` сверяет состояние на границе хода, внешний `fs/write` меняет файл на V2.
Baseline эпохи всё ещё содержит V1.

**Ожидаемое поведение.**
- Источник истины об изменении — **Codec-fingerprint** (`ContextSource.fingerprint()`), а **не таймстемп**.
  `FileCacheDecorator` при `fs/write` обязан `invalidate()` + опубликовать сигнал изменения в единый источник.
- `ContextReconciler.reconcile()` строит `ContextSnapshot` и через `diff()` обнаруживает несовпадение
  отпечатка источника файла. Возможные исходы по `ChangeState`:
  - изменение на безопасной границе хода → `UPDATED`, формируются `new_tail_messages` с новой версией,
    при необходимости `epoch_broken = True` (baseline пересобран — потеря prompt-cache, но **корректность важнее**);
  - изменение замечено внутри хода (небезопасно применять сейчас) → `DEFERRED`, применится на следующей границе.
- **Никогда** не должно быть состояния, где baseline содержит V1, а кэш/реальность — V2 без отметки
  (тихий рассинхрон — это баг, который прямо запрещён требованием интеграции Phase 2 ↔ Phase 4).

**Какой слой/компонент отвечает.** Слой B — `ContextReconciler`, `ContextSnapshot.diff()`, `ContextEpoch`;
слой C — `FileCacheDecorator` (`invalidate()` + единый сигнал), `FileContentCache`.

**Acceptance.**
- Тест: собрать baseline по V1, эмулировать `fs/write`→V2 (через декоратор), вызвать `reconcile()` →
  `ReconcileResult.state` ∈ {`UPDATED`,`DEFERRED`}, `updated_sources` содержит source_id файла.
- При `UPDATED` `new_tail_messages` (или новый baseline при `epoch_broken`) отражают V2, не V1.
- Нет ситуации `UNCHANGED` при фактически изменённом отпечатке (отсутствие тихого рассинхрона).
- `FileContentCache.get(path)` после `fs/write` не возвращает старое V2-содержимое как V1 (инвалидация сработала).

## Кейс 7 — Один файл прочитан и записан в одном ходе (порядок set/invalidate)

**Ситуация.** В рамках одного хода LLM делает `fs/read db.py` (декоратор кладёт содержимое в кэш),
затем `fs/write db.py` (новое содержимое). Оба tool-вызова проходят через `FileCacheDecorator`.

**Ожидаемое поведение.**
- Порядок операций детерминирован и привязан к порядку tool-исполнения:
  `fs/read` → `FileContentCache.set(path, content_v1)`; `fs/write` → `FileContentCache.invalidate(path)`.
- После `invalidate()` кэш по пути **пуст** (`get(path)` == `None`) — последующий `read` сходит к файлу заново
  и увидит V2. Запись **всегда** перекрывает предшествующее чтение в том же ходе.
- Защита от обратной гонки (отложенный `set` старого содержимого перезаписывает `invalidate`):
  для MVP — eventual consistency задокументирована, но в рамках **одного хода** порядок строго
  последовательный (tool-вызовы исполняются по очереди в `LLMLoopStage`), поэтому внутри хода гонки нет.
- `invalidate()` обязан опубликовать сигнал изменения, чтобы `ContextSnapshot` на границе хода увидел новый отпечаток.

**Какой слой/компонент отвечает.** Слой C — `FileCacheDecorator` (последовательность set/invalidate),
`FileContentCache.invalidate()`; стык со слоем B — единый сигнал → `ContextReconciler`.

**Acceptance.**
- Тест: `read`(set v1) → `write`(invalidate) в одном ходе → `FileContentCache.get(path)` == `None`.
- Следующий `read` в том же или следующем ходе возвращает V2 (новое содержимое из файла), не закэшированный V1.
- `invalidate()` вызван **после** `set()` (порядок проверяется на моке/спае декоратора).
- На границе хода `reconcile()` видит обновлённый отпечаток источника файла.

## Кейс 8 — Пустая история + пустой prompt

**Ситуация.** `build_context()` вызван с `session` без истории и `prompt = []` (или prompt из пустых блоков).
`TaskAnalyzer.analyze()` получает пустой текст.

**Ожидаемое поведение.**
- `build_context()` **не падает**: возвращает валидный `PayloadEnvelope` с `baseline`, содержащим
  только системный слой (system_prompt + стабильные источники, например `SkillContextSource`), и **пустым `tail`**.
- `TaskAnalyzer.analyze("")` возвращает безопасный дефолтный `TaskProfile`
  (например `task_type=FEATURE`, `search_terms=[]`, `investigation_depth=1`) — без LLM-вызова на пустой вход
  либо с детерминированным фолбэком.
- `ContextGatherer.gather()` при пустых `search_terms` не делает «слепой» обход проекта — возвращает `[]`
  (релевантного сбора нет, нечего собирать).
- `PayloadEnvelope.token_count` == токены baseline; `to_messages()` возвращает непустой системный префикс.

**Какой слой/компонент отвечает.** Вход — `ContextManager.build_context()`;
слой A — `TaskAnalyzer` (дефолтный профиль), `ContextGatherer` (пустой сбор без терминов).

**Acceptance.**
- `build_context(session=empty, prompt=[])` возвращает `PayloadEnvelope` без исключений.
- `envelope.tail == []`; `envelope.baseline` непуст (есть system_prompt).
- `gather()` с пустыми `search_terms` возвращает `[]` и не делает `search()`-RPC.
- `to_messages()` не пустой (по крайней мере system).

## Кейс 9 — Дубли файлов из разных источников (дедуп по id)

**Ситуация.** Один и тот же `db.py` отобран и `ContextGatherer` (как зависимость), и через
`SkillContextSource`/`ContextRegistry` (как часть скилла). На вход компактора/сборки приходят два
`ContextItem` с одинаковым `id` (= путь файла), возможно с разным `priority`.

**Ожидаемое поведение.**
- Дедуп по `ContextItem.id` (MVP-семантика как в FCM Option A: **overwrite / merge**, не исключение).
- При коллизии id выживает **один** элемент; правило слияния детерминировано:
  берётся **максимальный `priority`** (критичность не теряется), наиболее свежее `content`
  (последний записавший / больший `last_accessed`).
- В `baseline` файл попадает **ровно один раз** — иначе дублируется содержимое и рвётся стабильность
  отпечатка (повторяющийся текст раздувает токены и `baseline_fingerprint`).

**Какой слой/компонент отвечает.** Слой C — реестр `ContextItem` (dedup по `id`, merge priority);
слой A — `ContextRegistry` (источники не плодят дубли при render); вход — сборка `baseline`.

**Acceptance.**
- Добавление двух `ContextItem` с одинаковым `id` (priority 5, затем 7) → в результате один элемент с `priority == 7`.
- Содержимое итогового элемента — от последнего/наиболее свежего источника.
- В `envelope.baseline` нет двух сообщений с идентичным содержимым одного файла.
- `baseline_fingerprint` стабилен независимо от порядка прихода дублей (детерминированная сортировка перед хешированием).

## Кейс 10 — Переполнение после Prune+Skeletonize+Summarize

**Ситуация.** `ensure_context_fits()` запускает 3-фазный `ContextCompactor`
(Prune → Skeletonize → Summarize), но даже после всех трёх фаз
`token_count > max_context_tokens - reserved_tokens`. Типично: множество критических
(`priority >= 10`) элементов — `system_rules`, `user_prompt`, — которые **не вытесняются**.

**Ожидаемое поведение.**
- Сначала компактор отрабатывает все фазы по порядку; критические элементы не трогаются eviction'ом.
- Если после Summarize всё равно не влезает — **fail fast с явной диагностикой** (аналог FCM-кейса
  «critical items exceed budget»): исключение с перечислением критических `ContextItem.id` и их суммой токенов,
  предложением уменьшить критический контент или увеличить `max_context_tokens`.
- Это **конфигурационная/входная ошибка**, а не штатная ситуация: тихо ронять важное (system/user) нельзя.
- Деградация фаз: при недоступности LLM фаза Summarize пропускается (см. Кейс — деградация в §7 архитектуры),
  тогда переполнение тем вероятнее и приводит к тому же явному отказу, а не к зависанию.

**Какой слой/компонент отвечает.** Вход — `ContextManager.ensure_context_fits()`;
слой B/C — `ContextCompactor` (3 фазы), `ConversationSummarizer`, `TokenCounter` (финальная проверка).

**Acceptance.**
- Набор только из критических элементов суммой > бюджета → `ensure_context_fits()` поднимает `ValueError`
  с сообщением вида «critical items … exceed budget» и списком id.
- При наличии вытесняемых элементов те же входные данные проходят: после Prune/Skeletonize/Summarize итог <= лимита.
- При недоступном LLM компактор делает Prune+Skeletonize и, если не влезло, отказывает явно (не виснет).

## Кейс 11 — Эпоха разорвана несколько раз подряд (частые правки)

**Ситуация.** Пользователь/агент правит файлы каждый ход. Каждый `reconcile()` даёт
`ReconcileResult.epoch_broken = True` → baseline пересобирается → `baseline_fingerprint` меняется →
prompt-cache провайдера промахивается раз за разом.

**Ожидаемое поведение.**
- Корректность сохраняется: каждый разрыв даёт согласованный новый baseline (V_n). **Нет зацикливания**,
  нет бесконечного пересоздания эпох в рамках одного хода — разрыв происходит **не чаще одного раза за ход**
  (на безопасной границе).
- Деградация — **производительная, не функциональная**: prompt-cache хит падает к нулю, стоимость хода
  стремится к режиму «гидрация каждый ход» (Phase 1 baseline). Это допустимая нижняя граница, заложенная
  гибридным мостом (Phase 1 ≡ поведение гидрации за тем же API).
- Защита от деградации (опц., post-MVP): дебаунс мелких изменений через `ChangeState.DEFERRED` —
  серия правок схлопывается и применяется одной пересборкой на границе, а не каждый ход.
- Метрика разрывов эпохи: высокий процент `epoch_broken` за сессию → сигнал, что инкрементальность
  не даёт выгоды для этого паттерна (но не ошибка).

**Какой слой/компонент отвечает.** Слой B — `ContextReconciler` (частота разрывов, `DEFERRED`-дебаунс),
`ContextEpoch` (пересборка baseline), `ReconcileResult.epoch_broken`.

**Acceptance.**
- N ходов с правкой файла каждый ход → N корректных baseline, **без** роста числа эпох внутри одного хода (ровно ≤1 разрыв/ход).
- Цикл не зацикливается (тест завершается за конечное время), память не растёт неограниченно (старые эпохи освобождаются).
- При включённом дебаунсе серия правок в одном ходе даёт один `epoch_broken`, не несколько.
- Производительный фолбэк: при 100% `epoch_broken` стоимость ≈ режиму гидрации, результат корректен.

## Кейс 12 — Коллизия baseline_fingerprint / нестабильный скелет

**Ситуация.** Два **разных** baseline дают одинаковый `baseline_fingerprint` (коллизия хеша) —
тогда провайдер вернёт кэш чужого префикса. Либо обратное: `CodeSkeletonizer.skeletonize()` для
**одного и того же** входа даёт **разный** вывод (порядок методов, недетерминированный обход AST,
несортированные множества импортов) — тогда отпечаток «дрожит» и кэш промахивается на стабильном файле.

**Ожидаемое поведение.**
- **Детерминизм скелета — жёсткое требование** (зафиксировано в `CodeSkeletonizer.skeletonize()`:
  «один вход → один выход»). Реализация обязана: стабильный порядок узлов AST, отсортированные импорты,
  нормализованные пробелы/кавычки. Любая недетерминированность = промах кэша на ровном месте → дорогая регрессия.
- `baseline_fingerprint` — Codec-отпечаток над **каноникализованным** содержимым baseline
  (стабильный порядок сообщений и источников — см. дедуп в Кейсе 9). Используется
  криптостойкий хеш достаточной длины, чтобы коллизия была пренебрежимой; отпечаток **зависит от всего**
  содержимого baseline (а не от подмножества), чтобы разные baseline не схлопывались в один ключ.
- Почему критично: `FileContentCache` (детерминированное содержимое) + детерминированный скелет →
  файл в baseline байт-в-байт идентичен между ходами → стабильный префикс → кэш-хит. Любое звено
  недетерминизма рвёт всю цепочку экономии слоя B.

**Какой слой/компонент отвечает.** Слой C — `CodeSkeletonizer` (детерминизм вывода), `FileContentCache`
(детерминированное содержимое); слой B — расчёт `baseline_fingerprint` (`PayloadEnvelope`/`ContextEpoch`),
каноникализация baseline перед хешированием.

**Acceptance.**
- `skeletonize(code)` вызван 100 раз на одном входе → 100 идентичных строк (golden-тест на детерминизм).
- Перестановка порядка методов в исходнике, не меняющая семантику отбора, даёт стабильный (нормализованный) скелет.
- Идентичный baseline (тот же контент, любой порядок прихода источников) → идентичный `baseline_fingerprint`.
- Различающиеся baseline → различающиеся `baseline_fingerprint` (нет коллизии на тестовом корпусе).
- Регрессионный тест: один и тот же файл два хода подряд без правок → `baseline_fingerprint` не изменился (предпосылка кэш-хита).

## Кейс 13 — Скоуп с лимитом 0 или отрицательным

**Ситуация.** Конфиг или вызов задаёт `max_context_tokens = 0` (или отрицательное),
либо доли бюджета (`system_share` и т.п.) дают неположительную аллокацию для скоупа.

**Ожидаемое поведение.**
- **Fail fast при создании/валидации конфига** (аналог FCM-кейса negative max_tokens): `ContextConfig`
  и/или `TokenBudgetManager.allocate()` поднимают `ValueError` с понятным сообщением
  («max_context_tokens must be positive, got 0»). Это конфигурационная ошибка — падать рано и явно.
- `reserved_tokens >= max_context_tokens` (эффективно нулевой/отрицательный рабочий бюджет) — тоже отказ
  на валидации: рабочий бюджет `max_context_tokens - reserved_tokens` обязан быть положительным.
- Доли бюджета: сумма `system_share + history_share + tool_output_share + response_buffer_share`
  должна давать корректную (положительную, в пределах 1.0) аллокацию; иначе — ошибка валидации.

**Какой слой/компонент отвечает.** Конфиг — `ContextConfig` (валидация при загрузке TOML/env);
слой A — `TokenBudgetManager.allocate()` (проверка положительности рабочего бюджета).

**Acceptance.**
- `ContextConfig(max_context_tokens=0)` → `ValueError` при валидации.
- `max_context_tokens=-100` → `ValueError`.
- `reserved_tokens >= max_context_tokens` → `ValueError` (рабочий бюджет неположителен).
- Валидный конфиг (`128000`/`4096`) проходит и `allocate()` даёт все доли > 0.

## Кейс 14 — tool_result без соответствующего tool_call (orphaned)

**Ситуация.** В `session.history` оказался `tool_result`-блок без предшествующего `tool_call`
с тем же id (обрезан Prune'ом, рассинхрон истории, частичный сбой `LLMLoopStage`). Большинство
провайдеров отвергают такой payload (нарушение протокола tool-use).

**Ожидаемое поведение.**
- **Санитизация перед формированием `PayloadEnvelope`**: при сборке `tail`/`baseline` каждый `tool_result`
  сопоставляется с `tool_call` по id в пределах payload. Осиротевший `tool_result` **удаляется** (или сворачивается
  в нейтральное текстовое сообщение), чтобы не сломать протокол провайдера.
- Симметрично: `tool_call` без последующего `tool_result` (оборванный ход) — либо дополняется
  плейсхолдером-результатом, либо `tool_call` удаляется, чтобы пара была согласована.
- Фаза **Prune** компактора обязана резать **парами** (tool_call + его tool_result вместе), чтобы
  не плодить сирот при сжатии — это первичная профилактика; санитизация — страховка на входе.
- Факт санитизации логируется (`orphaned_tool_result_dropped` с id), без падения хода.

**Какой слой/компонент отвечает.** Слой B — `ContextCompactor` (Prune режет пары),
санитизация при сборке payload (вход `ContextManager.build_context()` / `ensure_context_fits()`);
инвариант разворачивания в плоский список — на стыке с `ExecutionEngine` (`to_messages()`).

**Acceptance.**
- История с `tool_result` без парного `tool_call` → итоговый `to_messages()` не содержит осиротевший блок (или он нейтрализован).
- `tool_call` без `tool_result` → пара согласована (плейсхолдер или удаление call).
- Prune на истории с tool-парами не оставляет сирот: после сжатия каждый `tool_result` имеет парный `tool_call`.
- Payload после санитизации принимается провайдером (контрактный тест на валидность tool-протокола).
- Лог фиксирует сброшенные осиротевшие блоки.
