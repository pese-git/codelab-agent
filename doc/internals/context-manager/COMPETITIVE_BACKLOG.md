## 2. Детальное описание идей

### 2.1. Персистентная память (MEMORY.md/USER.md)

**Источник:** Hermes Agent
**Описание:** Два файла с ограниченными лимитами (800 + 500 токенов), которые агент сам управляет через `memory` tool. Загружаются в system prompt как frozen snapshot в начале сессии.

**Ключевые особенности:**
- Агент сам решает, что сохранять (проактивно)
- Жёсткие лимиты предотвращают раздувание system prompt
- Frozen snapshot сохраняет prompt-cache хит
- Security scanning на prompt injection

**Ценность для пользователя:**
- Агент запоминает предпочтения, окружение, уроки между сессиями
- Не нужно повторять одно и то же каждый раз
- Персонализация без ручного конфигурирования

**Сложность реализации:**
- Новый слой E (Memory) — расширение архитектуры CM
- Интеграция с `ContextSource` (слой A)
- Tool для управления памятью (add/replace/remove)
- Security scanner для injection protection
- Механизм frozen snapshot (загрузка в начале, неизменность в сессии)

**Оценка:** 3-4 недели (1 разработчик)

**Зависимости:**
- Phase 0-3 должны быть завершены (стабильный baseline/tail API)
- Требуется отдельный ADR по архитектуре памяти

**Риски:**
- Может конфликтовать с инкрементальными эпохами (если memory меняется mid-session)
- Security scanner может блокировать легитимный контент (false positives)

### 2.2. Learning loop / авто-создание skills

**Источник:** Hermes Agent
**Описание:** Агент автоматически создаёт procedural skills после сложных задач. Skills само-улучшаются при повторном использовании.

**Ключевые особенности:**
- Триггер: завершение сложной задачи (много шагов, много tool calls)
- Генерация SKILL.md с описанием workflow
- Self-improvement: skill обновляется при повторном использовании
- Skills Hub для обмена (agentskills.io)

**Ценность для пользователя:**
- Агент учится на опыте, не повторяет ошибки
- Сложные workflow автоматизируются
- Knowledge sharing между пользователями

**Сложность реализации:**
- Детектор «сложных задач» (эвристики или LLM-классификация)
- Генератор skill-документов (LLM-based)
- Система версионирования skills
- Механизм self-improvement (diff-based updates)
- Интеграция с `SkillContextSource` (слой A)

**Оценка:** 4-6 недель (1 разработчик)

**Зависимости:**
- Персистентная память (2.1) — для хранения skills
- Phase 3 (ContextRegistry) — для рендеринга skills в baseline

**Риски:**
- Авто-создание может плодить низкокачественные skills
- Требуется механизм quality gate (review перед сохранением)

### 2.3. @-references для inline-инъекции

**Источник:** Hermes Agent
**Описание:** Синтаксис `@file:path`, `@diff`, `@git:5` для ручной инъекции контекста в сообщение пользователя.

**Ключевые особенности:**
- `@file:path[:line1-line2]` — содержимое файла или диапазон строк
- `@diff` / `@staged` — git diff
- `@git:N` — последние N коммитов
- `@url:...` — веб-контент
- Tab completion в CLI
- Size limits (soft 25%, hard 50% context window)

**Ценность для пользователя:**
- Быстрая инъекция контекста без tool calls
- Точный контроль над тем, что видит агент
- Удобство для code review (`@diff`)

**Сложность реализации:**
- Парсер @-синтаксиса в CLI/presentation layer
- Интеграция с `ContextGatherer` (слой A) — добавление items в tail
- Security: блокировка sensitive paths (SSH keys, credentials)
- Binary file detection

**Оценка:** 1 неделя (1 разработчик)

**Зависимости:**
- Phase 1 (ContextGatherer) — для добавления items
- CLI/presentation layer — для парсинга и tab completion

**Риски:**
- Может конфликтовать с автоматическим сбором (дублирование)
- Требует чёткой документации для пользователей

### 2.4. LSP-интеграция (diagnostics tool)

**Источник:** OpenCode CLI
**Описание:** Tool `diagnostics` для получения ошибок/предупреждений от Language Server Protocol.

**Ключевые особенности:**
- Multi-language support (Go, TypeScript, Python, ...)
- Получение ошибок без запуска кода
- Интеграция с существующими LSP-клиентами
- Автоматическое предложение фиксов

**Ценность для пользователя:**
- Агент видит ошибки компиляции/линтинга
- Может предлагать фиксы до запуска кода
- Уменьшение цикла edit-compile-test

**Сложность реализации:**
- Интеграция с существующим LSP-клиентом CodeLab
- Новый tool в `ToolRegistry`
- Форматирование diagnostics для LLM (компактное представление)
- Конфигурация language servers

**Оценка:** 2 недели (1 разработчик)

**Зависимости:**
- Существующий LSP-клиент в CodeLab
- Phase 1 (ToolRegistry) — для регистрации tool

**Риски:**
- LSP-серверы могут быть медленными (latency)
- Не все языки имеют качественные LSP-серверы

### 2.5. User modeling (Honcho dialectic)

**Источник:** Hermes Agent
**Описание:** Dialectic user modeling — агент строит модель пользователя через диалог, обновляет её по мере взаимодействия.

**Ключевые особенности:**
- Автоматическое извлечение предпочтений из диалога
- Обновление USER.md (или аналога) в фоне
- Dialectic approach: агент задаёт уточняющие вопросы
- Интеграция с Honcho (внешний провайдер)

**Ценность для пользователя:**
- Персонализация без явного конфигурирования
- Агент «понимает» стиль общения, уровень экспертизы
- Адаптация ответов под пользователя

**Сложность реализации:**
- Моделирование пользователя (data model)
- Детектор предпочтений в диалоге (LLM-based)
- Фоновый updater (background review)
- Интеграция с персистентной памятью (2.1)

**Оценка:** 3-4 недели (1 разработчик)

**Зависимости:**
- Персистентная память (2.1)
- Phase 3 (ConversationSummarizer) — для анализа диалога

**Риски:**
- Может быть invasif (пользователь не хочет, чтобы его моделировали)
- Требуется механизм consent (opt-in/opt-out)

### 2.6. Session search (FTS5)

**Источник:** Hermes Agent
**Описание:** Полнотекстовый поиск по прошлым сессиям (SQLite FTS5) с возможностью scroll внутри найденной сессии.

**Ключевые особенности:**
- FTS5 index по всем сообщениям
- Query: `session_search(query="...", session_id=...)`
- Scroll: `session_search(session_id=..., offset=...)`
- Хранение в SQLite (state.db)

**Ценность для пользователя:**
- Агент может вспомнить, что обсуждалось неделю назад
- Поиск по истории без ручного просмотра
- Восстановление контекста после перерыва

**Сложность реализации:**
- SQLite schema для sessions + messages
- FTS5 index creation/updates
- Tool `session_search` с query/scroll/browse
- Интеграция с storage layer

**Оценка:** 2 недели (1 разработчик)

**Зависимости:**
- Storage layer (SQLite)
- Phase 1 (ToolRegistry) — для регистрации tool

**Риски:**
- FTS5 может быть медленным на больших объёмах
- Требует миграции существующих данных (если storage не SQLite)

### 2.7. External memory providers

**Источник:** Hermes Agent
**Описание:** Плагины для внешней памяти: knowledge graphs, semantic search, automatic fact extraction.

**Ключевые особенности:**
- 8 провайдеров в Hermes (Honcho, Mem0, Hindsight, ...)
- Работают alongside built-in memory
- Разные возможности: semantic search, auto-extraction, user modeling
- Plugin architecture

**Ценность для пользователя:**
- Расширенные возможности памяти (semantic search, knowledge graphs)
- Выбор провайдера под задачу
- Интеграция с внешними системами

**Сложность реализации:**
- Plugin architecture для memory providers
- API для провайдеров (add/search/get)
- Интеграция с персистентной памятью (2.1)
- Конфигурация провайдеров

**Оценка:** 4-8 недель (1 разработчик)

**Зависимости:**
- Персистентная память (2.1)
- Plugin system в CodeLab

**Риски:**
- Зависимость от внешних сервисов (availability, cost)
- Сложность поддержки множества провайдеров

---

### 2.8. Progressive subdirectory discovery

**Источник:** Hermes Agent
**Описание:** Автоматическая загрузка AGENTS.md из поддиректорий по мере навигации агента.

**Ключевые особенности:**
- При чтении файла проверяются родительские директории
- AGENTS.md загружается и инжектируется в контекст
- Каждый поддиректорий проверяется один раз за сессию
- Security scan на prompt injection

**Ценность для пользователя:**
- Контекст появляется по мере необходимости (не раздувает system prompt)
- Поддержка monorepo (разные AGENTS.md для разных частей)
- Автоматизация (не нужно вручную прикладывать)

**Сложность реализации:**
- Hook на tool calls (read_file, terminal)
- Path extraction из аргументов
- Ancestor walk (до 5 уровней вверх)
- Интеграция с `ContextRegistry` (слой A)

**Оценка:** 1 неделя (1 разработчик)

**Зависимости:**
- Phase 1 (ContextRegistry) — для регистрации источников
- ToolRegistry — для hook на tool calls

**Риски:**
- Может замедлить tool calls (дополнительная проверка)
- Требует чёткой документации (какие AGENTS.md загружаются)

## 3. Связь с существующими фазами CM

### 3.1. Какие идеи можно добавить в Phase 6+

| Идея | Фаза | Обоснование |
|------|------|-------------|
| @-references (2.3) | Phase 6+ | Не требует изменений CM, только CLI/presentation |
| LSP-интеграция (2.4) | Phase 6+ | Новый tool, не требует изменений CM |

### 3.2. Какие идеи требуют отдельного ADR

| Идея | Обоснование |
|------|-------------|
| Персистентная память (2.1) | Новый слой E, расширение архитектуры |
| Learning loop (2.2) | Требует дизайна detector + generator |
| User modeling (2.5) | Требует дизайна модели + consent mechanism |
| Session search (2.6) | Требует выбора storage (SQLite vs существующий) |
| External providers (2.7) | Plugin architecture, интеграция с внешними системами |

### 3.3. Рекомендуемый порядок реализации

```
Phase 7 — Quick wins (2-3 недели)
  ├─ @-references (2.3) — 1 неделя

  └─ LSP-интеграция (2.4) — 2 недели (параллельно)

Phase 8 — Memory foundation (5-6 недель)
  ├─ Персистентная память (2.1) — 3-4 недели
  └─ Session search (2.6) — 2 недели (параллельно)

  ├─ Learning loop (2.2) — 4-6 недель

  └─ External providers (2.7) — 4-8 недель
```

## 4. Критерии отбора

### 4.1. Как оценивать идеи для включения

**Ценность для пользователя:**
- Решает реальную боль? (High value)
- Улучшает UX? (Medium value)
- Nice-to-have? (Low value)

**Сложность реализации:**
- Требует изменений архитектуры? (High complexity)
- Локализованные изменения? (Medium complexity)
- Добавление нового компонента? (Low complexity)

**Совместимость с текущей архитектурой:**
- Вписывается в слои A-D? (Low risk)
- Требует новый слой? (High risk)
- Конфликтует с существующими решениями? (Blocker)

### 4.2. Когда отклонять идею

- **Blocker:** Конфликт с ACP/MCP спецификациями
- **Blocker:** Нарушение обратной совместимости
- **High risk:** Требует переписывания ядра CM
- **Low value:** Не решает реальную боль пользователя

### 4.3. Когда включать идею

- **High value + Low complexity:** Быстрая победа, включать немедленно
- **High value + High complexity:** Требует отдельного ADR, планировать в roadmap
- **Medium value + Low complexity:** Включать по возможности (quick win)
- **Low value:** Отложить до востребованности
